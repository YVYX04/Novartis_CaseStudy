# The Limitations of Agentic AI in Industry

## 1. Introduction

Agentic AI should be understood not merely as a chatbot (LLM), but as a system that plans, reasons, retrieves context, and executes actions through tools. That form of agency is precisely what creates its industrial value, but it is also what makes its limitations extremely severe. In enterprise environments, the central problem is no longer only whether the model can generate plausible text; it is whether it can act correctly, consistently, safely, and accountably across long action chains inside real workflows. Recent enterprise-oriented benchmarks show that this threshold has not yet been reached. In CRMArena-Pro, leading generic agents reached only about 58% success in single-turn business tasks, with performance falling to roughly 35% in multi-turn settings; policy adherence and confidentiality awareness were particularly weak. In SCUBA, which evaluates computer-use agents on realistic Salesforce workflows, open-source agents scored below 5% in zero-shot settings, while stronger closed-source systems reached up to 39%, or about 50% with demonstrations ([Salesforce, 2025](https://www.salesforce.com/blog/why-generic-llm-agents-fall-short/), [Huang et al., 2025](https://doi.org/10.48550/arXiv.2505.18878)). These are failure rates one would probably not accept for business-critical operations. The following sections explore three main limitations of agentic AI in industry.

## 2. Operational Reliability

The first major limitation, therefore, is operational reliability. Agentic systems degrade as tasks become longer, more interactive, and more stateful. This is especially true when the agent must ask follow-up questions, decide which tool to use, recover from errors, or reconcile contradictory information. Anthropic’s own engineering guidance makes the point sharply: when a toolset is bloated or ambiguous, the agent’s decision quality deteriorates, because if a human cannot clearly determine which tool should be used or the very precise context in which the agent operates, the agent should not be expected to perform better. In practice, this means that many apparently impressive agent demos are artifacts of narrow, carefully selected task design rather than evidence of robust industrial autonomy. Finally, in entreprise settings where reliability is especially important since some decisions will be audited, the stochastic nature of the agents complexify the task of designing deterministic and repeatable business processes.

*Sources: [Salesforce, 2025](https://www.salesforce.com/blog/why-generic-llm-agents-fall-short/), [Anthropic, 2025](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents), and [Mohammadi et al., 2025](https://arxiv.org/html/2507.21504v1).*

## 2. Security

Once an agent can browse, execute code, modify files, access CRM systems, or connect to external servers, the threat model changes fundamentally. OpenClaw is a useful real-world illustration. Recent safety analyses describe vulnerabilities including prompt-injection-driven remote code execution, sequential tool attack chains, context amnesia, and supply-chain contamination. In another April 2026 study, poisoning a single persistent state dimension in OpenClaw raised average attack success rates from 24.6% to 64–74%; even the strongest tested defense still left a 63.8% success rate under one attack class. Anthropic’s own documentation likewise states that no system is completely immune to attacks and notes that third-party MCP servers are not managed or audited by Anthropic. The implication for industry is straightforward: the more capable the agent, the broader the attack surface and depth.

*Sources: [Ying et al., 2026](https://arxiv.org/abs/2603.12644) and [Wang et al., 2026](https://arxiv.org/abs/2604.04759).*

## 3. Data Structuration & Availability

A further and often underestimated limitation of agentic AI is that its effectiveness depends heavily on the prior organization of the firm’s data infrastructure. An agent cannot reason reliably over corporate knowledge if that knowledge is fragmented across siloed systems, poorly indexed, inconsistently labeled, or governed by ambiguous access rights. Major enterprise providers now treat this as a foundational issue rather than a secondary implementation detail: Microsoft explicitly frames enterprise "knowledge sources" as the basis on which agents can retrieve relevant information from internal systems, while Azure’s agentic retrieval architecture requires a defined knowledge base or indexed connection to authoritative sources. AWS makes the same point through retrieval-augmented generation, emphasizing that models must be connected to an authoritative and up-to-date knowledge base rather than rely on parametric memory alone. Anthropic’s recent work on context engineering likewise shows that agent performance deteriorates when relevant context is noisy, excessive, or poorly structured. In practical terms, this means that before deploying agentic AI, a company such as Novartis must invest in data cleaning, metadata discipline, document standardization, permissioning, and unified knowledge access; otherwise, the agent will simply automate confusion at scale rather than produce value.
